🚀 Kettle Master Pro - Guidelines Ingénierie & ArchitectureCe document définit les standards techniques et de design pour le développement de nos Web Apps (PWA) haute performance. En tant que Senior Dev / CTO, l'objectif est de produire un code Scalable, Sécurisé et User-Centric.🏗️ 1. Principes Fondamentaux de CodeNous appliquons les standards de l'industrie pour éviter la dette technique :SOLIDS : Single Responsibility (Une fonction ou un composant = une seule tâche).O : Open/Closed (Les composants sont ouverts à l'extension, fermés à la modification).L : Liskov Substitution (Les sous-types doivent être remplaçables par leur type de base).I : Interface Segregation (Préférer plusieurs petites interfaces à une seule massive).D : Dependency Inversion (Dépendre des abstractions, pas des implémentations).KISS (Keep It Simple, Stupid) & DRYNe sur-ingénierie pas. Factorisation des constantes et de la logique métier via des Hooks custom.🤖 2. IA & Agents Intelligents (Best Practices)L'intégration de l'IA (Gemini, OpenAI) doit suivre une architecture Server-Side Only pour garantir la sécurité et la performance.Architecture Proxy : Ne jamais appeler une API d'IA directement depuis le frontend. Passer systématiquement par une Cloud Function.Gestion des Secrets : Utilisation impérative de Google Cloud Secret Manager pour stocker les clés API. Interdiction formelle de les mettre dans les fichiers .env ou le code source.Streaming Response : Pour l'UX, privilégier le streaming des réponses de l'IA pour éviter l'effet "gel" de l'interface pendant le traitement.Prompt Engineering Versioning : Traiter les prompts comme du code. Les versionner et les stocker côté serveur pour pouvoir les mettre à jour sans redéployer le client.⚡ 3. Firebase Architecture & Cloud (Advanced)Authentication & SecurityApp Check : Activation de Firebase App Check pour protéger nos Cloud Functions et Firestore contre le spam et le vol de quota par des bots tiers.Claims Personnalisés : Utilisation des Custom Claims pour gérer les rôles (Admin, Pro, User) de manière sécurisée côté serveur.Cloud Functions (Node.js 2nd Gen)Concurrency : Utiliser les fonctions de 2ème génération pour gérer plusieurs requêtes simultanément sur une seule instance (réduction des coûts et des Cold Starts).Global Regions : Déployer les fonctions dans la région la plus proche des utilisateurs (ex: europe-west1).Idempotence : S'assurer que les fonctions de déclenchement (ex: onDocumentCreated) sont idempotentes pour éviter les erreurs en cas de retry automatique de Firebase.Firestore & StorageIndexation : Aucun tri ou filtrage complexe sans index composite pré-configuré.TTL (Time To Live) : Configurer des politiques TTL pour supprimer automatiquement les données temporaires (ex: logs de sessions IA) et réduire les coûts de stockage.🛠️ 4.
